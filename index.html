Chapter 1

15분

학습 목표

이 챕터를 끝내면 MCP가 무엇인지, 왜 굳이 "서버"라는 무서운 단어를 쓰는지, 그리고 이 가이드를 끝까지 읽으면 손에 무엇이 남는지를 알 수 있어요. 코드는 한 줄도 안 씁니다. 편하게 읽으세요.

30초 동기부여 — "내 알고리즘을 LLM이 직접 부르게 한다는 것"

지금 ChatGPT나 Claude 같은 챗봇에게 "내 데이터로 계산해줘"라고 부탁하면 어떻게 되나요? 보통은 데이터를 챗창에 붙여넣거나, 챗봇이 내놓은 답을 다시 우리가 해석해서 다음 질문을 던집니다. 챗봇은 우리의 도구를 직접 만지지 못합니다.

MCP(Model Context Protocol, 모델 컨텍스트 프로토콜)는 이 벽을 무너뜨리는 약속입니다. 한 번 만들어두면 Claude도, Cursor도, 다른 어떤 AI Agent든 똑같은 방식으로 내가 만든 도구를 직접 호출할 수 있어요. 마치 USB-C 단자처럼요. 모니터든 충전기든 USB-C만 있으면 꽂아 쓰듯이, AI도 MCP만 있으면 내 도구에 꽂혀서 쓸 수 있습니다.

이 가이드를 읽고 나면 손에 남는 것

(1) Claude Desktop에서 직접 호출되는 작은 MCP 서버 한 개, (2) 자기 알고리즘을 어떤 전략으로 MCP에 올릴지에 대한 의사결정, (3) "LLM이 잘 쓰는 도구"를 만드는 7~8가지 작은 습관. 코드는 처음부터 끝까지 우리가 같이 한 줄씩 씁니다.

MCP를 일상 비유로 풀기 — 식당 메뉴판

식당에 들어가면 메뉴판이 있죠. 메뉴판에는 "김치찌개 / 8,000원 / 매움 보통" 처럼 이름과 설명, 그리고 어떻게 주문해야 하는지가 적혀 있습니다. 손님은 메뉴판만 보고도 무엇을 시킬 수 있는지 알 수 있어요.

MCP 서버는 LLM에게 "메뉴판"을 건네주는 일을 합니다. 우리가 만든 함수에 이름과 설명, 입력 인자, 결과 형태를 적어두면, LLM은 그 메뉴판을 읽고 "아, 두 숫자를 더해주는 add라는 도구가 있구나" 하고 인식해서 직접 주문(호출)을 합니다. 실제 요리(계산)는 우리가 만든 코드가 합니다.

"프로토콜"이라는 단어가 어렵게 느껴지면

프로토콜은 그냥 "두 사람(여기서는 두 프로그램)이 서로 말이 통하도록 미리 정해둔 약속"이라는 뜻이에요. "안녕하세요 → 안녕하세요"가 인사 프로토콜이라면, MCP는 "이런 형식으로 도구 목록을 알려줘 → 이런 형식으로 도구를 호출해줘"라는 약속입니다.

"서버"라는 단어가 무서워 보이는 사람을 위한 5줄 설명

  • "서버"라고 하면 데이터센터에 있는 큰 컴퓨터가 떠오르지만, 여기서는 그냥 "요청을 받으면 응답하는 작은 프로그램 한 개" 라고 생각하세요.
  • 우리가 만들 MCP "서버"는 사실 Python 파일 한 개입니다. 길이도 20~30줄 정도예요.
  • 이 작은 프로그램이 내 PC 안에서 가만히 기다리고 있다가, Claude Desktop이 "두 숫자 더해줘"라고 부르면 응답합니다.
  • 외부에 공개되거나 인터넷에 노출되지 않습니다 (Chapter 7에서 다룰 stdio 방식 기준).
  • 그러니까 "서버를 만든다 = Python 파일 한 개를 만든다" 정도로 가볍게 생각하면 됩니다.

Tool / Resource / Prompt — MCP의 세 가지 도구함

MCP 서버는 LLM에게 세 종류의 "도구함"을 열어줄 수 있습니다. 셋의 차이를 한 표로 정리해 둡니다. 지금은 이름과 분위기만 기억해도 충분해요. 본격적으로 코드를 쓰는 건 Chapter 4입니다.

종류한 단어로무엇을 하나예시
Tool (도구)동사LLM이 "행동"을 하도록 시킬 때. 함수를 호출해 결과를 받음.add(a, b), search_papers(query), run_simulation(params)
Resource (자료)명사LLM이 "참고할 자료"를 읽어야 할 때. URL처럼 주소로 식별되는 읽기 전용 데이터.file://report.md, db://users/42
Prompt (프롬프트)양식사용자가 자주 쓰는 "질문 양식"을 미리 만들어 둘 때. 슬래시 명령처럼 호출./요약하기, /코드리뷰

이 가이드는 Tool에 집중합니다. 이유는 두 가지예요.

  1. 초보자의 1순위 동기는 "내 알고리즘/함수를 LLM이 호출하게 하기"인데, 그게 정확히 Tool의 역할이에요.
  2. Tool 한 가지만 익혀도 이미 충분히 강력합니다. Resource와 Prompt는 나중에 필요해지면 같은 방식으로 추가할 수 있어요.

LLM과 Agent는 뭐가 다른가요?

LLM(Large Language Model, 대형 언어 모델)은 GPT나 Claude처럼 글을 이해하고 만드는 모델 본체예요. Agent는 그 LLM이 도구를 직접 쓰게 한 형태입니다. "LLM이 손을 가지면 Agent가 된다"고 보면 됩니다. MCP는 그 손에 끼우는 장갑 같은 거고요.

이 가이드 끝에 만들 것 — 텍스트 시나리오

가이드를 다 끝낸 어느 평일 저녁의 풍경을 적어둘게요. 지금은 막연하게 들려도 Chapter 4가 끝나면 정확히 이 장면을 직접 경험합니다.

  1. Claude Desktop을 켭니다. 우하단에 작은 슬라이더 아이콘이 보입니다. 그게 "도구가 연결되어 있다"는 뜻이에요.
  2. "5와 7을 더해줘"라고 대화창에 입력합니다.
  3. Claude가 잠시 생각한 뒤 add 도구를 호출합니다. 우리가 짠 Python 함수가 진짜로 실행되어 12를 돌려줍니다.
  4. Claude는 그 12를 자기 답변에 자연스럽게 녹여서 "12입니다"라고 답합니다.
  5. 우리는 그 자리에서 한 번 더 웃은 뒤, add를 자기 알고리즘으로 바꿀 결심을 합니다.

옛날에는 가상환경을 직접 만들고 활성화해야 했지만(venv, pip...), uv는 이 모든 걸 알아서 합니다. 우리가 Chapter 4에서 uv add "mcp[cli]" 한 줄을 치면 uv가 알아서 가상환경을 만들고, 그 안에 mcp를 설치해줘요. 우리가 신경 쓸 건 없습니다.

안심 박스

"가상환경"이라는 단어를 보고 어렵다고 느낄 필요 없어요. 이 가이드 안에서는 단 한 번도 직접 만들지 않습니다. uv가 다 합니다.

핵심은 세 줄입니다.

  1. from my_algo import fibonacci — 기존 함수를 가져와서
  2. @mcp.tool() 데코레이터를 단 새 함수 compute_fibonacci를 만들고
  3. 그 안에서 return fibonacci(n)으로 위임

원래 함수의 docstring을 그대로 쓸 수도 있지만, MCP tool의 docstring은 LLM에게 보여줄 사용 설명서이기 때문에 더 풍부하게 다시 적는 걸 권합니다 (Chapter 9에서 이 주제를 본격적으로 다룹니다).

함수 시그니처 매핑 규칙

기존 Python 함수를 MCP tool로 옮길 때 챙겨야 할 것 네 가지:

  • 모든 인자에 타입 힌트 — n: int, name: str, items: list[int]. 타입 힌트가 없으면 LLM이 잘못된 형식으로 호출합니다.
  • 반환 타입도 표시 — -> int, -> dict. 가능하면 단순한 타입(숫자, 문자열, 리스트, 딕셔너리)으로.
  • JSON으로 직렬화 가능한 값만 반환 — numpy 배열, pandas DataFrame, 사용자 정의 클래스는 그대로 반환하면 안 돼요. 리스트나 딕셔너리로 변환해서 돌려주세요.
  • docstring에 "언제 쓰는지"를 적기 — 함수가 뭘 하는지 뿐만 아니라 LLM이 이 도구를 어떤 상황에서 골라야 하는지 한 줄을 추가하면 호출 정확도가 크게 올라갑니다.

numpy 배열을 받고 싶다면

가장 간단한 방법은 list[float]을 인자로 받고, 함수 안에서 np.array(values)로 변환하는 거예요. 반환할 때도 array.tolist()로 다시 리스트로 만들어 돌려줍니다. JSON 직렬화가 가능해야 LLM과 통신이 됩니다.

무거운 초기화는 한 번만

기존 알고리즘에 "큰 모델을 메모리에 올린다"거나 "수십 MB의 데이터 파일을 읽는다" 같은 무거운 초기화가 있다면, 그걸 함수 안에서 매번 하면 안 돼요. 호출할 때마다 모델을 다시 로딩하면 너무 느립니다.

해결책은 모듈 전역에서 1회만 로딩하는 거예요.

MODEL = joblib.load(...)은 서버 프로세스가 시작될 때 한 번만 실행됩니다. 이후의 모든 predict 호출은 이미 메모리에 있는 모델을 그대로 씁니다.

stdio 시작 timeout 함정

Claude Desktop은 서버를 시작한 뒤 일정 시간(보통 몇 초~몇십 초) 안에 응답이 없으면 timeout을 냅니다. 모델 로딩이 1분 이상 걸린다면 처음 시작에서 실패해요. 해결 방법:

  • 더 작은 모델로 바꾸거나
  • lazy loading: 모듈 전역이 아닌, 첫 호출 시점에 로드하고 캐시
  • 또는 기존 모델 서버를 따로 띄워두고 MCP 서버는 HTTP로 부르기 (시나리오 B의 옵션 2)

Lazy loading 패턴

get_model()이 처음 불릴 때만 로딩하고, 그 다음부터는 캐시된 객체를 돌려줍니다. 서버 시작은 빨라지고 첫 호출만 살짝 느려져요.

장시간 실행 작업은 어떻게?

함수 한 번 호출에 30초 이상 걸리는 작업이 있다면, 일단 동기적으로 처리해보세요. MCP 클라이언트(Claude Desktop)는 호출이 끝날 때까지 기다려 줍니다.

문제는 사용자가 진행 상황을 알 수 없다는 거예요. 1분 동안 로딩 화면만 보면 답답하죠. 진행률(progress) 보고를 띄우는 방법이 있는데, 이건 Chapter 6에서 "MCP-friendly 설계 원칙"으로 다시 다룹니다. 지금은 "긴 작업도 일단은 동작은 한다"고만 알아두세요.

MCP-friendly 7가지 원칙 — 한 줄씩

  1. 순수 함수 지향 — 같은 입력은 같은 출력. 가능하면 전역 상태에 의존하지 않기.
  2. JSON 직렬화 가능한 입출력 — int, float, str, bool, list, dict만 받고 돌려주기.
  3. 무거운 초기화와 호출의 분리 — 모델 로딩 같은 건 모듈 전역 또는 lazy 캐시로 1회만.
  4. 명확한 에러 메시지 — 무엇이 잘못됐는지 + 왜 + 어떻게 고치면 되는지 세 가지를 한 문장에.
  5. 진행률 보고 가능성 — 30초 이상 걸리는 함수는 진행률을 띄울 수 있게 설계.
  6. 부작용 표시 — 데이터를 읽기만 하는지(readonly), 바꾸는지(mutate)를 docstring에 명시.
  7. 단일 책임 — 한 함수는 한 가지 일만. 너무 많은 일을 한 함수가 겸직하면 LLM이 호출 타이밍을 헷갈림.

각 원칙을 짧게 풀어 설명할게요.

원칙 1·2 — 순수 함수와 직렬화

순수 함수란 "외부 상태를 건드리지 않고, 같은 입력에는 같은 출력을 돌려주는" 함수예요. MCP에서 순수 함수가 좋은 이유는, LLM이 같은 도구를 여러 번 호출해도 결과가 예측 가능하기 때문입니다.

JSON 직렬화 가능한 타입만 사용하라는 건 Chapter 5에서도 말했죠. 한 번 더 정리하면:

  • OK: int, float, str, bool, list, dict, None
  • 변환 필요: numpy.ndarray → list, pandas.DataFrame → list[dict], datetime → ISO 8601 문자열
  • 권장하지 않음: 사용자 정의 클래스를 그대로 반환

Pydantic을 쓰면 자동으로 검증·직렬화 됩니다

입출력이 복잡하면 pydantic.BaseModel을 정의해서 인자나 반환값으로 쓰세요. MCP SDK가 자동으로 JSON Schema를 만들어 LLM에 보여줍니다. 자세한 사용법은 Chapter 9에서 다뤄요.

원칙 3 — 무거운 초기화와 호출 분리

Chapter 5에서 본 바로 그 패턴입니다. 모듈 전역에서 1회만 로드하거나, lazy loading 함수를 만들거나. 다시 한 번 강조하면:

  • 함수 안에서 매번 model = load_model(...)을 부르면 안 됩니다. 매 호출마다 느려져요.
  • 모듈 전역에 두면 서버 시작이 느려지지만 호출은 빨라요.
  • Lazy loading은 서버 시작은 빠르지만 첫 호출이 느려요.
  • 어떤 게 맞는지는 모델 크기와 사용 패턴에 따라 달라요.

원칙 4 — 명확한 에러 메시지

LLM은 도구가 던지는 에러 메시지를 읽고 다음에 어떻게 호출할지 결정합니다. 그래서 에러 메시지에는 세 가지가 들어가야 해요.

  • 왜 잘못됐는지 (원인)
  • 어떻게 고치면 되는지 (다음 행동)

긴 작업(예: 이미지 100장 처리)을 짤 때는 진행률을 보고할 수 있게 함수를 설계해두세요. MCP에는 진행률 알림(progress notification) 기능이 있어서, 함수에서 Context를 받으면 진행률을 업데이트할 수 있어요.

지금 자세히 외울 필요는 없어요. 핵심은 "새로 함수를 짤 때 인자에 진행률을 보낼 자리를 미리 만들어 두면 나중에 추가하기 쉽다" 는 점입니다.

async def가 처음 보일 때

Python의 비동기 함수 문법이에요. def 대신 async def로 쓰면 함수 안에서 await를 쓸 수 있고, 진행률 보고처럼 시간이 걸리는 작업을 더 잘 처리할 수 있습니다. MCP의 Context 객체는 비동기로 동작하기 때문에 async def가 필요해요. 지금은 "이런 게 있다"만 알아두고, 실제로 쓰게 되면 그때 더 깊이 배우면 됩니다.

원칙 6 — 부작용 표시 (readonly vs mutate)

도구가 데이터를 읽기만 하는지, 바꾸는지를 docstring에 분명히 적어두세요. LLM이 위험한 작업을 신중하게 다루도록 도와줍니다.

MCP SDK 1.x에는 도구에 메타데이터를 다는 방법(annotations)도 있지만, 가장 효과가 좋은 건 docstring에 명확히 적는 것입니다. LLM이 가장 먼저 읽는 게 docstring이거든요.

원칙 7 — 단일 책임

한 함수가 너무 많은 일을 하면 안 됩니다.

나쁜 예: process_data(action: str, data: dict)처럼 action에 따라 분기하는 함수. LLM은 어느 action을 골라야 할지 헷갈립니다.

  • 엔드포인트 한 개 (보통 /mcp)에 클라이언트가 POST로 요청을 보냄

(레거시) SSE — 동작 원리 30초 SSE (레거시)

SSE는 Streamable HTTP 이전의 원격 전송 방식이에요. 구조가 더 복잡합니다.

  • 두 개의 엔드포인트 가 있어요. GET /sse로 장기 SSE 스트림을 열고, 별도의 POST 엔드포인트로 클라이언트 메시지를 보냄.
  • 서버리스 환경(Vercel, Cloudflare Workers 등)에서 장기 연결을 유지하기 어려움
  • 로드 밸런서가 있으면 sticky session 설정이 필요
  • 세션 관리, 재전송 같은 기능이 내장되어 있지 않음

이런 단점들 때문에 2025-03-26 스펙에서 Streamable HTTP로 대체됐어요.

새 서버를 만든다면 SSE를 쓰지 마세요

Streamable HTTP가 거의 모든 면에서 우월합니다. SSE 섹션이 이 가이드에 있는 이유는 단 하나, 인터넷에서 검색해 나오는 옛 예제들이 아직 SSE 기준인 경우가 많기 때문 이에요. 그 코드들을 읽고 이해할 수 있어야 해서 짧게 짚고 넘어갑니다.

팁 1·2 — 네이밍과 Docstring (Before / After)

Before — 좋지 않은 버전

문제점:

  • 이름 calc가 너무 추상적이에요. LLM이 "이게 무슨 계산을 한다는 거지?" 하고 헷갈립니다.
  • 인자 이름 x, y는 의미가 없음. 타입 힌트도 없음.
  • docstring "do calculation"은 정보가 없는 거나 마찬가지.

After — 다듬은 버전

개선점:

  • 이름이 동사+명사(add_integers). "정수 두 개를 더한다"는 게 이름만 봐도 명확해요.
  • 첫 줄에 무엇을 하는지 한 문장.
  • 두 번째 단락에 언제 써야 하는지 + 두 개 이상이면 어떻게 해야 하는지.
  • 각 인자 설명에 타입 + 의미 + 예시.
  • 반환값 설명에 타입 + 의미 + 예시.
  • 부작용 표시.

LLM은 이 docstring 전체를 읽고 호출 결정을 합니다. 단어 하나하나가 모두 정확도에 영향을 줘요.

docstring을 길게 쓰는 게 정말 효과 있나요?

네. 실제로 같은 함수에 docstring만 짧은 버전 / 자세한 버전 두 개를 만들어 비교하면, 자세한 버전 쪽에서 LLM이 잘못 호출하는 비율이 눈에 띄게 줍니다. 단, 너무 길면 토큰을 낭비하므로 핵심 정보(언제 쓰는지, 인자 의미, 부작용)에 집중하세요.

팁 3 — 스키마: Pydantic Field

타입 힌트만으로도 충분히 동작하지만, 인자에 더 풍부한 메타데이터를 붙이고 싶다면 Pydantic Field를 씁니다. MCP SDK가 자동으로 JSON Schema로 변환해 LLM에 전달해줘요.

Annotated[int, Field(...)]이 처음 보면 어렵지만 패턴은 단순해요. 타입을 Annotated로 감싸고, 두 번째 자리에 Field를 적는다. 그 안의 description과 examples가 LLM이 보는 스키마에 그대로 포함됩니다.

Annotated가 뭔가요?

Python의 표준 typing 모듈에서 제공하는 "타입 힌트에 부가 정보를 붙이는 방법"이에요. Annotated[int, ...]는 "기본 타입은 int이고, 추가로 이런 메타데이터가 붙어 있다"는 뜻입니다. Pydantic이 이걸 읽어서 검증과 스키마를 만듭니다.

PyPI는 Python 패키지 공식 저장소예요. pip install hello-mcp 한 줄로 누구나 우리 서버를 설치할 수 있게 만드는 옵션입니다.

  • 다른 패키지의 의존성으로 들어가야 할 때

PyPI 업로드는 uv build → uv publish 두 명령으로 가능해요. 자세한 절차는 uv 공식 문서를 참고하세요.

Claude Desktop이 아닌 다른 클라이언트도 같은 방식

이 가이드는 Claude Desktop을 기준으로 진행했지만, MCP는 표준이라서 다른 클라이언트도 거의 똑같은 방식으로 등록됩니다.

  • Cursor: Settings → MCP → Add MCP server. JSON 구조는 Claude Desktop과 거의 동일.
  • Claude Code (VS Code 확장): 설정에서 MCP 서버 등록.
  • VS Code 빌트인 MCP 지원: 최근 버전에서 추가됨.

한 번 만든 서버를 여러 클라이언트에 동시에 등록할 수 있어요. 어떤 클라이언트에 있든 같은 도구가 같은 결과를 돌려줍니다. 이게 MCP가 표준인 이유의 본질이에요.
