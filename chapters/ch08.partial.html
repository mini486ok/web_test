이 챕터에서 중요한 사실 하나를 먼저 분명히 해둡니다. MCP의 현재 공식 표준 전송 방식은 stdio와 Streamable HTTP 두 가지입니다. 과거에는 SSE(Server-Sent Events)가 원격 전송이었지만, 2025-03-26 스펙 이후 Streamable HTTP로 대체되었고 SSE는 deprecated 상태예요. 이 가이드도 그 사실을 그대로 따라서, Streamable HTTP를 권장 원격 전송으로 다루고 SSE는 "기존 자료를 읽을 때를 위한 레거시 호환" 으로만 짧게 다룹니다.

Streamable HTTP — 동작 원리 30초 Streamable HTTP

Streamable HTTP는 한 줄로 말하면 단일 HTTP 엔드포인트 하나로 양방향 통신을 처리하는 방식 입니다.

  • 엔드포인트 한 개 (보통 /mcp)에 클라이언트가 POST로 요청을 보냄
  • 서버는 응답을 일반 JSON으로 줄 수도 있고, 더 긴 응답이면 Server-Sent Events 스트림으로 줄 수도 있음
  • 클라이언트가 GET으로 같은 엔드포인트를 두드리면 서버가 알림(notification)을 푸시할 수 있는 SSE 스트림이 열림
  • 세션은 Mcp-Session-Id 헤더로 관리, 끊겼다 다시 붙으면 Last-Event-ID로 재개 가능

복잡해 보이지만 우리가 직접 짤 일은 없어요. FastMCP가 다 처리해 줍니다. 우리는 한 글자만 바꾸면 돼요.

0.0.0.0으로 바인딩하는 순간 같은 네트워크의 다른 PC가 우리 MCP 서버를 호출할 수 있습니다. 기본 상태에서는 인증이 없다는 점을 꼭 기억하세요. 다음 세 가지를 같이 챙기지 않으면 절대 인터넷에 노출하지 마세요.

  1. Origin 헤더 검증 — DNS rebinding 공격 방지. MCP 스펙의 MUST 조항.
  2. 인증 — 토큰 기반 또는 OAuth. 프로덕션에서는 필수.
  3. HTTPS — Cloudflare Tunnel, nginx reverse proxy 등으로 TLS 종단.

이 가이드에서는 깊이 다루지 않습니다. 친구와 같은 LAN에서 잠깐 테스트하는 정도라면 LAN 내부에서만 사용하고, 진짜 인터넷 노출은 추가 학습이 필요합니다.

(레거시) SSE — 동작 원리 30초 SSE (레거시)

SSE는 Streamable HTTP 이전의 원격 전송 방식이에요. 구조가 더 복잡합니다.

  • 두 개의 엔드포인트 가 있어요. GET /sse로 장기 SSE 스트림을 열고, 별도의 POST 엔드포인트로 클라이언트 메시지를 보냄.
  • 서버리스 환경(Vercel, Cloudflare Workers 등)에서 장기 연결을 유지하기 어려움
  • 로드 밸런서가 있으면 sticky session 설정이 필요
  • 세션 관리, 재전송 같은 기능이 내장되어 있지 않음

이런 단점들 때문에 2025-03-26 스펙에서 Streamable HTTP로 대체됐어요.

새 서버를 만든다면 SSE를 쓰지 마세요

Streamable HTTP가 거의 모든 면에서 우월합니다. SSE 섹션이 이 가이드에 있는 이유는 단 하나, 인터넷에서 검색해 나오는 옛 예제들이 아직 SSE 기준인 경우가 많기 때문 이에요. 그 코드들을 읽고 이해할 수 있어야 해서 짧게 짚고 넘어갑니다.

"stdio랑 streamable-http 두 개를 동시에 띄울 수 있나요?"

같은 코드 파일을 두 번 실행하면 됩니다. 한 번은 stdio로 (Claude Desktop이 띄우게), 다른 한 번은 Streamable HTTP로 (직접 PowerShell에서 띄워서 다른 도구가 붙게). 같은 코드가 두 가지 통로를 동시에 처리할 수 있다는 게 FastMCP의 좋은 점이에요.
