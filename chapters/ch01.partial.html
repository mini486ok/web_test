Chapter 1

MCP가 뭐고 왜 만드는가

15분

학습 목표

  • MCP가 무엇인지 한 문장으로 설명할 수 있다
  • 내가 가진 알고리즘을 MCP로 감싸야 하는 이유를 한 가지 말할 수 있다
  • Tool / Resource / Prompt 세 가지가 각각 무슨 일을 하는지 구분할 수 있다

이 챕터를 끝내면 MCP가 무엇인지, 왜 굳이 "서버"라는 무서운 단어를 쓰는지, 그리고 이 가이드를 끝까지 읽으면 손에 무엇이 남는지를 알 수 있어요. 코드는 한 줄도 안 씁니다. 편하게 읽으세요.

30초 동기부여 — "내 알고리즘을 LLM이 직접 부르게 한다는 것"

지금 ChatGPT나 Claude 같은 챗봇에게 "내 데이터로 계산해줘"라고 부탁하면 어떻게 되나요? 보통은 데이터를 챗창에 붙여넣거나, 챗봇이 내놓은 답을 다시 우리가 해석해서 다음 질문을 던집니다. 챗봇은 우리의 도구를 직접 만지지 못합니다.

MCP(Model Context Protocol, 모델 컨텍스트 프로토콜)는 이 벽을 무너뜨리는 약속입니다. 한 번 만들어두면 Claude도, Cursor도, 다른 어떤 AI Agent든 똑같은 방식으로 내가 만든 도구를 직접 호출할 수 있어요. 마치 USB-C 단자처럼요. 모니터든 충전기든 USB-C만 있으면 꽂아 쓰듯이, AI도 MCP만 있으면 내 도구에 꽂혀서 쓸 수 있습니다.

이 가이드를 읽고 나면 손에 남는 것

(1) Claude Desktop에서 직접 호출되는 작은 MCP 서버 한 개, (2) 자기 알고리즘을 어떤 전략으로 MCP에 올릴지에 대한 의사결정, (3) "LLM이 잘 쓰는 도구"를 만드는 7~8가지 작은 습관. 코드는 처음부터 끝까지 우리가 같이 한 줄씩 씁니다.

MCP를 일상 비유로 풀기 — 식당 메뉴판

식당에 들어가면 메뉴판이 있죠. 메뉴판에는 "김치찌개 / 8,000원 / 매움 보통" 처럼 이름과 설명, 그리고 어떻게 주문해야 하는지가 적혀 있습니다. 손님은 메뉴판만 보고도 무엇을 시킬 수 있는지 알 수 있어요.

MCP 서버는 LLM에게 "메뉴판"을 건네주는 일을 합니다. 우리가 만든 함수에 이름과 설명, 입력 인자, 결과 형태를 적어두면, LLM은 그 메뉴판을 읽고 "아, 두 숫자를 더해주는 add라는 도구가 있구나" 하고 인식해서 직접 주문(호출)을 합니다. 실제 요리(계산)는 우리가 만든 코드가 합니다.

"프로토콜"이라는 단어가 어렵게 느껴지면

프로토콜은 그냥 "두 사람(여기서는 두 프로그램)이 서로 말이 통하도록 미리 정해둔 약속"이라는 뜻이에요. "안녕하세요 → 안녕하세요"가 인사 프로토콜이라면, MCP는 "이런 형식으로 도구 목록을 알려줘 → 이런 형식으로 도구를 호출해줘"라는 약속입니다.

"서버"라는 단어가 무서워 보이는 사람을 위한 5줄 설명

  • "서버"라고 하면 데이터센터에 있는 큰 컴퓨터가 떠오르지만, 여기서는 그냥 "요청을 받으면 응답하는 작은 프로그램 한 개" 라고 생각하세요.
  • 우리가 만들 MCP "서버"는 사실 Python 파일 한 개입니다. 길이도 20~30줄 정도예요.
  • 이 작은 프로그램이 내 PC 안에서 가만히 기다리고 있다가, Claude Desktop이 "두 숫자 더해줘"라고 부르면 응답합니다.
  • 외부에 공개되거나 인터넷에 노출되지 않습니다 (Chapter 7에서 다룰 stdio 방식 기준).
  • 그러니까 "서버를 만든다 = Python 파일 한 개를 만든다" 정도로 가볍게 생각하면 됩니다.

안심 박스

Chapter 4까지만 따라오면 누구나 첫 MCP 서버를 만들고 Claude Desktop에서 호출까지 마칩니다. 지금 "내가 그런 거 만들 수 있을까?" 싶은 분도 30분만 같이 손을 움직이면 됩니다.

Tool / Resource / Prompt — MCP의 세 가지 도구함

MCP 서버는 LLM에게 세 종류의 "도구함"을 열어줄 수 있습니다. 셋의 차이를 한 표로 정리해 둡니다. 지금은 이름과 분위기만 기억해도 충분해요. 본격적으로 코드를 쓰는 건 Chapter 4입니다.

종류한 단어로무엇을 하나예시
Tool (도구)동사LLM이 "행동"을 하도록 시킬 때. 함수를 호출해 결과를 받음.add(a, b), search_papers(query), run_simulation(params)
Resource (자료)명사LLM이 "참고할 자료"를 읽어야 할 때. URL처럼 주소로 식별되는 읽기 전용 데이터.file://report.md, db://users/42
Prompt (프롬프트)양식사용자가 자주 쓰는 "질문 양식"을 미리 만들어 둘 때. 슬래시 명령처럼 호출./요약하기, /코드리뷰

이 가이드는 Tool에 집중합니다. 이유는 두 가지예요.

  1. 초보자의 1순위 동기는 "내 알고리즘/함수를 LLM이 호출하게 하기"인데, 그게 정확히 Tool의 역할이에요.
  2. Tool 한 가지만 익혀도 이미 충분히 강력합니다. Resource와 Prompt는 나중에 필요해지면 같은 방식으로 추가할 수 있어요.

LLM과 Agent는 뭐가 다른가요?

LLM(Large Language Model, 대형 언어 모델)은 GPT나 Claude처럼 글을 이해하고 만드는 모델 본체예요. Agent는 그 LLM이 도구를 직접 쓰게 한 형태입니다. "LLM이 손을 가지면 Agent가 된다"고 보면 됩니다. MCP는 그 손에 끼우는 장갑 같은 거고요.

이 가이드 끝에 만들 것 — 텍스트 시나리오

가이드를 다 끝낸 어느 평일 저녁의 풍경을 적어둘게요. 지금은 막연하게 들려도 Chapter 4가 끝나면 정확히 이 장면을 직접 경험합니다.

  1. Claude Desktop을 켭니다. 우하단에 작은 슬라이더 아이콘이 보입니다. 그게 "도구가 연결되어 있다"는 뜻이에요.
  2. "5와 7을 더해줘"라고 대화창에 입력합니다.
  3. Claude가 잠시 생각한 뒤 add 도구를 호출합니다. 우리가 짠 Python 함수가 진짜로 실행되어 12를 돌려줍니다.
  4. Claude는 그 12를 자기 답변에 자연스럽게 녹여서 "12입니다"라고 답합니다.
  5. 우리는 그 자리에서 한 번 더 웃은 뒤, add를 자기 알고리즘으로 바꿀 결심을 합니다.

이게 이 가이드의 핵심 성공 경험이에요.
