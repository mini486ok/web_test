시나리오 A — 이미 Python으로 구현됨

가장 쉬운 길입니다. 큰 그림은 한 줄이에요: 기존 함수를 import해서 새로 만든 얇은 wrapper 함수에 @mcp.tool()을 붙이면 끝.

예시 상황

가정: 내 PC에 이미 잘 동작하는 알고리즘이 있다고 합시다. 편의상 my_algo.py라는 파일에 함수 한 개가 있다고 해볼게요.

C:\mcp-lab\hello-mcp\my_algo.py:

(여러분의 실제 알고리즘은 이거보다 훨씬 복잡하겠지만, 구조는 똑같습니다.)

매핑하기

Chapter 4에서 만든 server.py에 다음을 추가해 봅시다.

핵심은 세 줄입니다.

  1. from my_algo import fibonacci — 기존 함수를 가져와서
  2. @mcp.tool() 데코레이터를 단 새 함수 compute_fibonacci를 만들고
  3. 그 안에서 return fibonacci(n)으로 위임

원래 함수의 docstring을 그대로 쓸 수도 있지만, MCP tool의 docstring은 LLM에게 보여줄 사용 설명서이기 때문에 더 풍부하게 다시 적는 걸 권합니다 (Chapter 9에서 이 주제를 본격적으로 다룹니다).

함수 시그니처 매핑 규칙

기존 Python 함수를 MCP tool로 옮길 때 챙겨야 할 것 네 가지:

  • 모든 인자에 타입 힌트 — n: int, name: str, items: list[int]. 타입 힌트가 없으면 LLM이 잘못된 형식으로 호출합니다.
  • 반환 타입도 표시 — -> int, -> dict. 가능하면 단순한 타입(숫자, 문자열, 리스트, 딕셔너리)으로.
  • JSON으로 직렬화 가능한 값만 반환 — numpy 배열, pandas DataFrame, 사용자 정의 클래스는 그대로 반환하면 안 돼요. 리스트나 딕셔너리로 변환해서 돌려주세요.
  • docstring에 "언제 쓰는지"를 적기 — 함수가 뭘 하는지 뿐만 아니라 LLM이 이 도구를 어떤 상황에서 골라야 하는지 한 줄을 추가하면 호출 정확도가 크게 올라갑니다.

numpy 배열을 받고 싶다면

가장 간단한 방법은 list[float]을 인자로 받고, 함수 안에서 np.array(values)로 변환하는 거예요. 반환할 때도 array.tolist()로 다시 리스트로 만들어 돌려줍니다. JSON 직렬화가 가능해야 LLM과 통신이 됩니다.

무거운 초기화는 한 번만

기존 알고리즘에 "큰 모델을 메모리에 올린다"거나 "수십 MB의 데이터 파일을 읽는다" 같은 무거운 초기화가 있다면, 그걸 함수 안에서 매번 하면 안 돼요. 호출할 때마다 모델을 다시 로딩하면 너무 느립니다.

해결책은 모듈 전역에서 1회만 로딩하는 거예요.

MODEL = joblib.load(...)은 서버 프로세스가 시작될 때 한 번만 실행됩니다. 이후의 모든 predict 호출은 이미 메모리에 있는 모델을 그대로 씁니다.

stdio 시작 timeout 함정

Claude Desktop은 서버를 시작한 뒤 일정 시간(보통 몇 초~몇십 초) 안에 응답이 없으면 timeout을 냅니다. 모델 로딩이 1분 이상 걸린다면 처음 시작에서 실패해요. 해결 방법:

  • 더 작은 모델로 바꾸거나
  • lazy loading: 모듈 전역이 아닌, 첫 호출 시점에 로드하고 캐시
  • 또는 기존 모델 서버를 따로 띄워두고 MCP 서버는 HTTP로 부르기 (시나리오 B의 옵션 2)

Lazy loading 패턴

get_model()이 처음 불릴 때만 로딩하고, 그 다음부터는 캐시된 객체를 돌려줍니다. 서버 시작은 빨라지고 첫 호출만 살짝 느려져요.

장시간 실행 작업은 어떻게?

함수 한 번 호출에 30초 이상 걸리는 작업이 있다면, 일단 동기적으로 처리해보세요. MCP 클라이언트(Claude Desktop)는 호출이 끝날 때까지 기다려 줍니다.

문제는 사용자가 진행 상황을 알 수 없다는 거예요. 1분 동안 로딩 화면만 보면 답답하죠. 진행률(progress) 보고를 띄우는 방법이 있는데, 이건 Chapter 6에서 "MCP-friendly 설계 원칙"으로 다시 다룹니다. 지금은 "긴 작업도 일단은 동작은 한다"고만 알아두세요.

C/C++, Rust, Java, MATLAB, C# 등으로 구현된 알고리즘을 MCP에 올리는 길은 세 가지가 있어요. 각각의 장단점을 한 표로 봅시다.

옵션난이도성능유지보수언제 고르나
1. CLI 래퍼쉬움중간좋음이미 .exe나 CLI 도구가 있다
2. REST/HTTP 래퍼중간좋음매우 좋음무거운 초기화가 있고, 기존 언어로 서버를 띄울 수 있다
3. 네이티브 바인딩어려움매우 좋음어려움호출이 매우 잦고 성능이 핵심이다

옵션 1: CLI 래퍼 (가장 쉬움)

기존 알고리즘이 이미 명령줄에서 동작하는 실행 파일(.exe, .jar, Rust 바이너리 등)이라면, Python의 표준 모듈 subprocess로 그냥 호출만 하면 됩니다.

핵심 인자 네 개:

  • capture_output=True — 도구의 출력을 메모리에 잡아둔다
  • text=True — bytes 대신 문자열로 받는다
  • encoding="utf-8" — Windows의 기본 인코딩(cp949) 때문에 한글이 깨지는 걸 방지
  • check=True — 도구가 0이 아닌 종료 코드를 반환하면 예외를 던진다

한글 출력이 깨질 때

Windows에서 subprocess를 쓰면 한글이 깨지는 일이 정말 흔합니다. 위처럼 encoding="utf-8"을 명시하세요. 그래도 안 되면 외부 도구가 한글 출력을 cp949로 내보내고 있을 가능성이 있어요. 도구 쪽 출력 인코딩을 UTF-8로 바꾸거나, encoding="cp949"로 받은 다음 처리하세요.

옵션 2: REST/HTTP 래퍼

기존 알고리즘이 이미 (또는 새로) HTTP 서버를 띄울 수 있다면, MCP 서버는 그냥 HTTP 클라이언트가 됩니다. 이 방법은 무거운 모델을 한 번만 로딩하고 여러 호출을 빠르게 처리할 수 있어요.

준비물: httpx 패키지를 추가합니다.

이 패턴의 강점:

  • 모델 로딩이 MCP 서버 시작 시간과 분리됨 → Claude Desktop의 timeout 회피
  • 기존 알고리즘 서버를 다른 클라이언트(웹 앱, CI 등)와 공유 가능
  • 알고리즘을 다시 컴파일/배포할 때 MCP 서버를 건드릴 필요가 없음

옵션 3: 네이티브 바인딩

매 호출마다 프로세스를 시작하는 오버헤드가 부담이고(예: 호출이 초당 수십 번), 성능이 정말 핵심이라면 네이티브 바인딩을 고려합니다. 단, 난이도가 가장 높습니다.

언어별 추천 도구:

  • C / C++: ctypes (Python 표준 라이브러리), cffi
  • Rust: PyO3 기반으로 Python 패키지를 직접 빌드
  • Java: JPype, Py4J
  • C# / .NET: pythonnet
  • MATLAB: MATLAB Engine API for Python

이 방식은 빌드 환경이 까다롭고, 한 번 빌드한 패키지를 다른 PC로 옮길 때 OS/아키텍처가 같아야 한다는 제약이 있어요. 처음 MCP를 만든다면 옵션 1이나 2부터 시도하는 걸 강력히 권합니다.

시나리오 B 결정 가이드

  • 기존에 동작하는 .exe나 CLI 도구가 있고, 호출 빈도가 낮다 → 옵션 1 (CLI 래퍼)
  • 무거운 초기화가 있고, 기존 언어로 HTTP 서버를 띄우는 게 쉽다 → 옵션 2 (REST 래퍼)
  • 호출이 매우 잦고 성능 요구가 극단적이다 → 옵션 3 (네이티브 바인딩), 단 처음이면 옵션 2부터
