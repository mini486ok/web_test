Chapter 9

LLM이 잘 쓰는 MCP 만드는 8가지 팁

18분
선행 CH4, CH6

학습 목표

  • 도구 이름·docstring·스키마·에러 메시지를 LLM 친화적으로 다듬을 수 있다
  • Pydantic Field로 인자 description과 examples를 추가할 수 있다
  • Chapter 4의 add를 8가지 원칙을 모두 적용한 버전으로 다시 쓴다

도구가 동작하는 것과 LLM이 그 도구를 정확하게 사용하는 것은 다른 문제예요. 이 챕터에서는 같은 add 함수를 8가지 작은 습관을 적용해 단계적으로 다듬습니다. 각 팁이 왜 효과 있는지 한 줄씩 설명할게요. 18분 정도 걸려요.

8가지 팁 한눈에

#팁한 줄로
1도구 네이밍동사 + 명사. 너무 추상적이거나 너무 일반적이지 않게
2DocstringLLM이 가장 먼저 읽는 사용 설명서. 언제 쓸지·입력 의미·출력 의미
3입출력 스키마Pydantic Field로 description, examples 추가
4에러 메시지무엇이 / 왜 / 어떻게 고치나 세 가지
5도구 수 관리보통 5~15개. 너무 많으면 LLM이 헷갈림
6부작용 표시readonly vs mutate를 docstring에 명시
7예시 값description에 구체 예시 한 개
8반복 개선LLM이 잘못 호출하는 걸 보고 description 다듬기

팁 1·2 — 네이밍과 Docstring (Before / After)

Before — 좋지 않은 버전

문제점:

  • 이름 calc가 너무 추상적이에요. LLM이 "이게 무슨 계산을 한다는 거지?" 하고 헷갈립니다.
  • 인자 이름 x, y는 의미가 없음. 타입 힌트도 없음.
  • docstring "do calculation"은 정보가 없는 거나 마찬가지.

After — 다듬은 버전

개선점:

  • 이름이 동사+명사(add_integers). "정수 두 개를 더한다"는 게 이름만 봐도 명확해요.
  • 첫 줄에 무엇을 하는지 한 문장.
  • 두 번째 단락에 언제 써야 하는지 + 두 개 이상이면 어떻게 해야 하는지.
  • 각 인자 설명에 타입 + 의미 + 예시.
  • 반환값 설명에 타입 + 의미 + 예시.
  • 부작용 표시.

LLM은 이 docstring 전체를 읽고 호출 결정을 합니다. 단어 하나하나가 모두 정확도에 영향을 줘요.

docstring을 길게 쓰는 게 정말 효과 있나요?

네. 실제로 같은 함수에 docstring만 짧은 버전 / 자세한 버전 두 개를 만들어 비교하면, 자세한 버전 쪽에서 LLM이 잘못 호출하는 비율이 눈에 띄게 줍니다. 단, 너무 길면 토큰을 낭비하므로 핵심 정보(언제 쓰는지, 인자 의미, 부작용)에 집중하세요.

팁 3 — 스키마: Pydantic Field

타입 힌트만으로도 충분히 동작하지만, 인자에 더 풍부한 메타데이터를 붙이고 싶다면 Pydantic Field를 씁니다. MCP SDK가 자동으로 JSON Schema로 변환해 LLM에 전달해줘요.

Annotated[int, Field(...)]이 처음 보면 어렵지만 패턴은 단순해요. 타입을 Annotated로 감싸고, 두 번째 자리에 Field를 적는다. 그 안의 description과 examples가 LLM이 보는 스키마에 그대로 포함됩니다.

Annotated가 뭔가요?

Python의 표준 typing 모듈에서 제공하는 "타입 힌트에 부가 정보를 붙이는 방법"이에요. Annotated[int, ...]는 "기본 타입은 int이고, 추가로 이런 메타데이터가 붙어 있다"는 뜻입니다. Pydantic이 이걸 읽어서 검증과 스키마를 만듭니다.

더 복잡한 입출력엔 BaseModel

여러 필드를 가진 객체를 주고받을 때는 Pydantic BaseModel을 정의합니다.

LLM은 Point와 DistanceResult의 description까지 모두 받아 보기 때문에 호출 정확도가 크게 올라가요.

팁 5 — 도구 수 관리

한 서버에 도구가 30개, 50개씩 있으면:

  • LLM이 도구 목록을 읽는 데 토큰을 많이 쓰고
  • 비슷한 이름의 도구들 사이에서 헷갈리고
  • 응답 속도가 느려집니다.

경험적으로 한 서버에 5~15개가 적당해요. 너무 많아지면:

  • 비슷한 도구를 묶어 한 도구로 정리 (인자로 분기)
  • 또는 서버 자체를 두 개로 나누기 (예: read_*만 모은 서버, write_*만 모은 서버)

"인자로 분기"가 단일 책임 원칙(Chapter 6)과 충돌하지 않나요?

네, 실제로 충돌합니다. 그래서 균형을 잡아야 해요. 가이드라인:

  • 동작이 비슷하고 입력만 약간 다르면 → 한 도구로 묶기
  • 동작 자체가 다르면 → 도구를 따로 만들기 정답은 없고, LLM이 잘못 호출하는 패턴을 보면서 다듬는 게 가장 정확합니다.

팁 6·7 — 부작용 표시 + 예시 값

이미 위의 docstring 예시에서 자연스럽게 들어갔어요. 한 번 더 정리하면:

  • Side effects 섹션을 docstring에 추가. None이면 명시적으로 None이라고 적기. 파일/DB/네트워크에 변화를 만든다면 분명히 써두기.
  • Example 값을 인자와 반환값 모두에 적기. 추상적인 설명보다 구체적인 숫자/문자열 한 개가 더 도움이 됩니다.

팁 8 — 반복 개선 루프

마지막 팁이 가장 중요해요. 한 번에 완벽한 docstring을 쓸 수는 없습니다. 실제로 Claude/Cursor에서 자기 도구를 호출시켜 보고, LLM이 잘못 호출하는 패턴을 관찰하면서 description을 다듬는 게 정석이에요.

루프는 이렇게 돌아갑니다:

  1. 도구를 만들고 docstring을 일단 적당히 적는다
  2. Claude Desktop에서 자연어로 5~10번 호출해본다
  3. LLM이 헷갈리는 지점을 발견한다 (잘못된 인자, 잘못된 타이밍, 에러 후 멈춤 등)
  4. 그 지점에 맞춰 docstring 또는 에러 메시지를 다듬는다
  5. 다시 2번으로

이 루프를 한두 번 돌리는 것만으로도 도구 사용 정확도가 크게 올라갑니다.
