Chapter 6

처음부터 MCP를 염두에 둔 알고리즘 설계

15분
선행 CH5

학습 목표

  • MCP-friendly 7가지 설계 원칙을 한 줄씩 말할 수 있다
  • core 레이어와 server 레이어를 분리한 디렉토리 구조를 그릴 수 있다
  • 잘못된 함수 시그니처를 MCP-friendly로 다듬을 수 있다

새 알고리즘을 짤 때 미리 알아두면 나중에 고칠 일이 줄어드는 7가지 원칙입니다. 코드를 새로 짜는 게 아니라 머릿속 체크리스트를 만드는 챕터예요. 15분 정도면 충분합니다.

MCP-friendly 7가지 원칙 — 한 줄씩

  1. 순수 함수 지향 — 같은 입력은 같은 출력. 가능하면 전역 상태에 의존하지 않기.
  2. JSON 직렬화 가능한 입출력 — int, float, str, bool, list, dict만 받고 돌려주기.
  3. 무거운 초기화와 호출의 분리 — 모델 로딩 같은 건 모듈 전역 또는 lazy 캐시로 1회만.
  4. 명확한 에러 메시지 — 무엇이 잘못됐는지 + 왜 + 어떻게 고치면 되는지 세 가지를 한 문장에.
  5. 진행률 보고 가능성 — 30초 이상 걸리는 함수는 진행률을 띄울 수 있게 설계.
  6. 부작용 표시 — 데이터를 읽기만 하는지(readonly), 바꾸는지(mutate)를 docstring에 명시.
  7. 단일 책임 — 한 함수는 한 가지 일만. 너무 많은 일을 한 함수가 겸직하면 LLM이 호출 타이밍을 헷갈림.

각 원칙을 짧게 풀어 설명할게요.

원칙 1·2 — 순수 함수와 직렬화

순수 함수란 "외부 상태를 건드리지 않고, 같은 입력에는 같은 출력을 돌려주는" 함수예요. MCP에서 순수 함수가 좋은 이유는, LLM이 같은 도구를 여러 번 호출해도 결과가 예측 가능하기 때문입니다.

JSON 직렬화 가능한 타입만 사용하라는 건 Chapter 5에서도 말했죠. 한 번 더 정리하면:

  • OK: int, float, str, bool, list, dict, None
  • 변환 필요: numpy.ndarray → list, pandas.DataFrame → list[dict], datetime → ISO 8601 문자열
  • 권장하지 않음: 사용자 정의 클래스를 그대로 반환

Pydantic을 쓰면 자동으로 검증·직렬화 됩니다

입출력이 복잡하면 pydantic.BaseModel을 정의해서 인자나 반환값으로 쓰세요. MCP SDK가 자동으로 JSON Schema를 만들어 LLM에 보여줍니다. 자세한 사용법은 Chapter 9에서 다뤄요.

원칙 3 — 무거운 초기화와 호출 분리

Chapter 5에서 본 바로 그 패턴입니다. 모듈 전역에서 1회만 로드하거나, lazy loading 함수를 만들거나. 다시 한 번 강조하면:

  • 함수 안에서 매번 model = load_model(...)을 부르면 안 됩니다. 매 호출마다 느려져요.
  • 모듈 전역에 두면 서버 시작이 느려지지만 호출은 빨라요.
  • Lazy loading은 서버 시작은 빠르지만 첫 호출이 느려요.
  • 어떤 게 맞는지는 모델 크기와 사용 패턴에 따라 달라요.

원칙 4 — 명확한 에러 메시지

LLM은 도구가 던지는 에러 메시지를 읽고 다음에 어떻게 호출할지 결정합니다. 그래서 에러 메시지에는 세 가지가 들어가야 해요.

  • 무엇이 잘못됐는지 (사실)
  • 왜 잘못됐는지 (원인)
  • 어떻게 고치면 되는지 (다음 행동)

Before / After

나쁜 예:

b=0일 때 그냥 ZeroDivisionError가 던져집니다. LLM은 어떤 인자가 문제였는지 알기 어려워요.

좋은 예:

LLM은 이 에러 메시지를 받아서 "아, b를 0이 아닌 값으로 다시 호출해야겠다"고 자체적으로 판단할 수 있어요.

원칙 5 — 진행률 보고 가능성

긴 작업(예: 이미지 100장 처리)을 짤 때는 진행률을 보고할 수 있게 함수를 설계해두세요. MCP에는 진행률 알림(progress notification) 기능이 있어서, 함수에서 Context를 받으면 진행률을 업데이트할 수 있어요.

지금 자세히 외울 필요는 없어요. 핵심은 "새로 함수를 짤 때 인자에 진행률을 보낼 자리를 미리 만들어 두면 나중에 추가하기 쉽다" 는 점입니다.

async def가 처음 보일 때

Python의 비동기 함수 문법이에요. def 대신 async def로 쓰면 함수 안에서 await를 쓸 수 있고, 진행률 보고처럼 시간이 걸리는 작업을 더 잘 처리할 수 있습니다. MCP의 Context 객체는 비동기로 동작하기 때문에 async def가 필요해요. 지금은 "이런 게 있다"만 알아두고, 실제로 쓰게 되면 그때 더 깊이 배우면 됩니다.

원칙 6 — 부작용 표시 (readonly vs mutate)

도구가 데이터를 읽기만 하는지, 바꾸는지를 docstring에 분명히 적어두세요. LLM이 위험한 작업을 신중하게 다루도록 도와줍니다.

MCP SDK 1.x에는 도구에 메타데이터를 다는 방법(annotations)도 있지만, 가장 효과가 좋은 건 docstring에 명확히 적는 것입니다. LLM이 가장 먼저 읽는 게 docstring이거든요.

원칙 7 — 단일 책임

한 함수가 너무 많은 일을 하면 안 됩니다.

나쁜 예: process_data(action: str, data: dict)처럼 action에 따라 분기하는 함수. LLM은 어느 action을 골라야 할지 헷갈립니다.

좋은 예: analyze_data, clean_data, summarize_data처럼 동작별로 함수를 나눠 등록. 각 함수의 docstring이 명확해서 LLM이 정확히 고를 수 있어요.

그렇다고 도구 수가 너무 많아도 안 돼요

한 서버에 도구가 30개, 50개씩 있으면 LLM이 토큰을 너무 많이 써서 헷갈립니다. 보통 5~15개가 적당해요. Chapter 9에서 더 자세히 다룹니다.

디렉토리 구조 — core / server / tests

알고리즘이 조금만 커지면 한 파일에 다 넣지 말고 폴더를 나누세요. 알고리즘 코어와 MCP 어댑터를 분리하는 게 핵심이에요.

이렇게 분리하면 좋은 점:

  • 알고리즘 코어는 MCP 없이도 테스트할 수 있다. 그냥 Python 함수로 import해서 단위 테스트 작성 가능.
  • MCP 어댑터는 얇게 유지된다. core를 import해서 docstring과 타입 힌트만 신경쓰면 끝.
  • 나중에 다른 인터페이스(웹 API, CLI 등)도 같은 core를 재사용할 수 있다.
